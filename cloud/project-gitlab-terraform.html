GitLab CI로 AWS Terraform 자동 배포

2025 | CI/CD 실습

프로젝트 개요

GitLab CE와 GitLab Runner를 구축하여 Terraform 코드의 자동화된 검증 및 배포 파이프라인을 구성했습니다. Merge Request(MR) 기반 워크플로우를 통해 코드 변경사항을 검증하고 자동으로 AWS 인프라에 배포하는 CI/CD 프로세스를 구현했습니다.

구성 요소

GitLab CE

  • 자체 호스팅 GitLab Community Edition 설치
  • 프로젝트 저장소 및 Merge Request 기능 활용
  • CI/CD 파이프라인 설정 및 모니터링

GitLab Runner

  • Docker executor 기반 GitLab Runner 설치 및 등록
  • Terraform 실행 환경을 위한 Docker 이미지 구성
  • 병렬 실행을 위한 여러 Runner 구성 가능

CI/CD 파이프라인

  • validate: Terraform 코드 문법 검증
  • plan: 실행 계획 생성 및 리뷰
  • apply: 인프라 변경사항 적용 (메인 브랜치 병합 시)

파이프라인 워크플로우

1. 개발 브랜치에서 작업

  • 기능 브랜치 생성 및 Terraform 코드 수정
  • Push 시 자동으로 validate 및 plan 단계 실행
  • Plan 결과를 Merge Request에 코멘트로 자동 첨부

2. Merge Request 생성

  • 코드 리뷰를 위한 MR 생성
  • CI 파이프라인에서 변경사항 검증
  • Plan 단계에서 인프라 변경 영향도 확인

3. 메인 브랜치 병합

  • MR 승인 후 메인 브랜치로 병합
  • 자동으로 apply 단계 실행
  • AWS 인프라 변경사항 적용

AWS 연동

  • AWS IAM Role: GitLab Runner에서 사용할 IAM 역할 구성
  • S3 Backend: Terraform remote state를 S3에 저장
  • 환경 변수: GitLab CI/CD Variables를 통한 민감한 정보 관리
  • State Locking: DynamoDB를 통한 동시 실행 방지

보안 및 모범 사례

  • AWS 자격 증명을 환경 변수로 관리 (GitLab CI/CD Variables)
  • Terraform state 파일 암호화 (S3 server-side encryption)
  • Plan 단계에서 변경사항 사전 검토 필수
  • Protected branches를 통한 메인 브랜치 보호

학습 목표 및 성과

  • GitLab CI/CD 파이프라인을 통한 인프라 자동화 경험
  • Terraform과 CI/CD 통합을 통한 DevOps 워크플로우 이해
  • Docker 기반 Runner를 통한 실행 환경 격리 및 관리
  • AWS IAM 및 보안 모범 사례 적용
  • Infrastructure as Code와 CI/CD의 결합을 통한 효율적인 인프라 관리
